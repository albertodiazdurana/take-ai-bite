# Mirror Sync Manifest

**Version:** 1.0
**Created:** 2026-04-18
**Origin:** Mirror Sync Manifest initiative.
**Scope:** Authoritative categorized list of file types that mirror repos (currently TAB) receive from DSM Central.
**Companions:**
- [document-structure-standard.md](document-structure-standard.md)
- [skill-file-structural-standard.md](skill-file-structural-standard.md)

This manifest defines what gets mirror-synced and why. It exists so the
sync process cannot silently decay: categories are explicit, new files
have a visible home, and the follow-up bidirectional check script can
validate the actual sync list against this manifest.

## 1. Row Schema

Each in-scope category row has five fields:

- **Category:** short human-readable label.
- **Pattern:** glob or explicit file list describing the files in this category.
- **Examples:** 1-3 representative filenames drawn from current Central content.
- **Rationale:** one sentence on why mirrors need this category.
- **Mapping:** `1:1` by default. `source >> target` when the file's location on the mirror differs from Central (source >> target mapping syntax).

## 3. Out-of-Scope Items

Files Central has but does NOT mirror, with reasoning:

| Item | Reason not mirrored |
|------|---------------------|
| `MEMORY.md` (auto-memory path) | Per-instance. Each project has its own session history; mirroring would cross-contaminate identity. |
| `.claude/session-transcript.md`, `.claude/session-baseline.txt`, `.claude/parallel-sessions.txt`, `.claude/commit-lock` | Runtime ephemera. Session-scoped state; gitignored on Central; irrelevant on mirrors. |
| `.claude/last-align.txt`, `.claude/last-align-report.md`, `.claude/last-wrap-up.txt`, `.claude/kickoff-done.txt` | Per-instance local state markers. Mirrors generate their own on first session. |
| `.claude/CLAUDE.md`, `.claude/dsm-ecosystem.md`, `.claude/reasoning-lessons.md`, `.claude/skills-registry.md` (runtime paths, not `.template` copies) | Per-instance. Each project's runtime config is shaped by its own Kick-off or hand-authoring. The `.template` companions ARE mirrored (category 9). |
| `plan/` (Central backlog organization) | Central-only. Mirrors do not track Central's backlog; each project manages its own plans in `dsm-docs/plans/`. |
| `dsm-docs/` subfolders other than `guides/` (e.g., `handoffs/`, `checkpoints/`, `research/`, `plans/`, `decisions/`, `blog/`, `feedback-to-dsm/`) | Per-project governance artifacts. Each repo generates its own; mirroring would overwrite local work. |
| `scripts/commands/` runtime deploy copies on Central (`.claude/commands/*.md`) | Regenerated by `sync-commands.sh --deploy`; gitignored on Central. Mirrors track their deploy copies per category 6 via `source >> target` mapping. |
| `.claude/personal-rules-allowlist.md`, `.claude/contributor-profile.md` | Author-personal content (personal content scanner scope). Must never propagate. |

## 4. Maintenance Notes

### When to add a new category

A new category earns a manifest row when a new kind of file is routinely added in Central that mirrors need. Criteria:

- Files of this kind appear repeatedly (not a one-off).
- Mirrors break or degrade if the files are absent.
- The files share a stable naming pattern that a glob can match.

If any criterion fails, the file is either a one-off entry in an existing category, or belongs in section 3 (out-of-scope).

### How new files get added to the sync list

Today (2026-04-18), adding a new file that belongs to an existing category is a manual edit to `scripts/take-ai-bite-sync.txt`. This is the silent-decay risk the manifest exists to mitigate: the manifest declares intent, but enforcement is still manual.

The bidirectional check (follow-up to the manifest initiative) mechanizes the check: a bidirectional validator that (a) verifies every sync-list entry matches a manifest category and a real file, and (b) flags any actual tracked file matching a manifest pattern that is absent from the sync list. Run at `/dsm-version-update` Step 9 and `/dsm-wrap-up` mirror sync.

Until the bidirectional check script ships, the discipline is:
1. When creating a file in Central that matches a manifest category, pair the creation with a `take-ai-bite-sync.txt` entry in the same commit.
2. When adding a new manifest category (per the criteria above), update this file AND `take-ai-bite-sync.txt` AND `.claude/CLAUDE.md` Version Update Workflow Step 9 if the category warrants explicit mention.

### When to remove a category

If an entire category becomes obsolete (e.g., Kick-off templates retired in favor of a different bootstrap mechanism), remove the row here AND the corresponding section in `take-ai-bite-sync.txt`. Note the removal in the manifest's changelog section (to be added when first removal occurs).

### Versioning

This manifest is versioned by modification date in the header. Breaking changes to the row schema (e.g., adding a new required field) warrant a version bump (1.0 â†’ 2.0). Adding, removing, or editing category rows is routine maintenance at 1.0; no version bump needed.
